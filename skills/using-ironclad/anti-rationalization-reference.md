# Anti-Rationalization Reference

Central reference for all skills. If you're reading this, a skill directed you here because you may be rationalizing.

## The Pattern

Rationalization follows a predictable pattern:
1. You encounter a process requirement
2. You generate a reason why THIS case is different
3. The reason feels logical and productive
4. You skip the requirement
5. Quality suffers

**If you're generating reasons to skip a process step, that IS the signal to follow it.**

## Universal Red Flags

These thoughts — in ANY skill context — mean STOP:

| Thought | Reality |
|---------|---------|
| "Just this once" | The exception that becomes the rule |
| "Too simple to need this" | Simple is where assumptions hide |
| "I already know what to do" | Knowing ≠ following the process |
| "This is overkill" | Discipline prevents mistakes |
| "I'll do it after" | After never comes |
| "The spirit matters, not the ritual" | The ritual IS the spirit — it prevents shortcuts |
| "I can be pragmatic here" | Pragmatic = follow the process. Shortcuts = debugging later |
| "This is different because..." | It's not. Follow the process |
| "I already manually verified" | Manual ≠ systematic. No record, can't re-run |
| "Keep as reference" | You'll adapt instead of rebuilding. Delete means delete |

## Skill-Specific Triggers

**TDD:** "Code before test" → Delete code. Start over.
**Debugging:** "Try a quick fix" → Follow the 4-phase process.
**Brainstorming:** "Too simple for a design" → Every project gets a design.
**Skills:** "I remember this skill" → Skills evolve. Read current version.
**Review:** "Looks fine to me" → Use the checklist. Every time.
**Verification:** "Tests pass, we're done" → Run the full verification checklist.

## The Test

Ask yourself: "Would I be comfortable if my human partner watched me skip this step?"

If no → follow the process.
